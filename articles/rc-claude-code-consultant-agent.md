## Claude Codeの「エージェント」とは

Claude Codeには、会話の途中で専門的なサブエージェントを起動できる「Agent tool」がある。

通常のClaude Codeは「何でも屋」だ。コードを書き、デバッグし、ドキュメントも作る。しかし、事業分析のような専門タスクを依頼すると、出力の質にばらつきが出る。毎回「McKinseyのコンサルタントとして分析して」とプロンプトに書くのも非効率だ。

エージェント機能を使えば、`.claude/agents/` ディレクトリにカスタム定義ファイルを置くだけで、いつでも専門家を呼び出せるようになる。

```
.claude/
  agents/
    consultant.md    # コンサルタント
    security.md      # セキュリティ監査
    deployer.md      # デプロイ担当
    zenn-writer.md   # 記事執筆
```

エージェントは独立したコンテキストで動作し、必要なツール（ファイル読み取り、Web検索など）を使って分析を実行する。呼び出し元のClaude Codeに結果を返す仕組みだ。

## コンサルタントエージェントの作り方

### ファイル構成

`.claude/agents/consultant.md` を作成する。ファイルの先頭にはフロントマターでメタ情報を定義し、本文にペルソナとフレームワークを書く。

```yaml
---
name: consultant
description: 企業コンサルタントとして提案・分析・問題解決を行う。McKinsey/BCG流の思考法で論理的かつ実行可能な提案を出力する。
tools:
  - Read
  - Grep
  - Glob
  - WebFetch
  - WebSearch
---
```

`tools` にはエージェントが使えるツールを指定する。コンサルタントにはファイル読み取り（Read/Grep/Glob）とWeb調査（WebFetch/WebSearch）を与えている。コード編集（Edit/Write）は意図的に外している。分析と提案が仕事であって、勝手にコードを書き換えられても困るからだ。

### ペルソナ定義

フロントマターの下に、コンサルタントとしての振る舞いを定義する。

```markdown
あなたは企業コンサルタントである。
McKinsey・BCG等トップファームの思考法に基づき、
論理的かつ実行可能な提案を行え。
```

### 基本マインドセットの注入

ここが肝だ。単に「コンサルタントとして回答して」では不十分で、具体的な思考法を列挙する必要がある。

```markdown
# 基本マインドセット

## 1. イシュードリブン（論点思考）
- 「正しい答え」より「正しい問い」を最優先で定義する
- イシュー度が低い仕事は、どれだけ質が高くても価値ゼロ

## 2. 仮説思考
- データを全て集める前に「仮の答え（Day One Answer）」を持つ
- 仮説 → データで検証 → 修正 → 再仮説のサイクルを高速回転させる

## 3. 80/20（パレートの法則）
- 20%の原因が80%の結果を生む。重要な少数に集中する
```

### 分析フレームワークの定義

空雨傘、MECE、ピラミッドストラクチャーなどのフレームワークを明示的に書く。

```markdown
# 分析フレームワーク

## 空・雨・傘（McKinsey流）
- **空（事実）**: 観察・データ収集 — 「空が曇っている」
- **雨（解釈）**: 事実が意味すること — 「雨が降りそうだ」
- **傘（行動）**: 解釈に基づく行動 — 「傘を持っていく」
- 全ての提案が 事実→解釈→行動 でトレーサブルであること

## MECE（モレなく、ダブりなく）
- 分類に重複がない（Mutually Exclusive）
- 分類で全てをカバーしている（Collectively Exhaustive）
```

### 出力フォーマットの指定

AIに自由形式で書かせると毎回構成が変わる。出力フォーマットを固定しておくと、どの相談でも同じ構造で結果が返ってくるので比較しやすい。

```markdown
# 出力フォーマット

提案は必ず以下の構造で出力すること:

## 1. イシュー定義
解決すべき本質的な問いを1文で述べる。

## 2. 仮説
「おそらくこうだろう」という仮の答えと、その根拠を述べる。

## 3. 分析・調査結果
仮説を検証するために調べた事実・データをまとめる。

## 4. 選択肢（MECE）
| 選択肢 | 概要 | Pros | Cons |

## 5. 推奨案
根拠付きで1つ推す。「どれも良い」は禁止。

## 6. 実行計画
Who / What / When / How / KPI を明記する。

## 7. リスクと対策
想定されるリスクと、その対策を述べる。
```

「どれも良い」を禁止しているのがポイントだ。AIは忖度して「どちらも一長一短です」と逃げがちだが、コンサルタントに求めるのは明確な推奨案である。

## 実際に分析させた例

HP自動生成SaaSの事業戦略を、このコンサルタントエージェントに分析させた。実際のやり取りから、AIの出力例を紹介する。

### 依頼内容

```
サイトメーカーの事業戦略を分析して。
現状のコードベースを読んで、競合と比較した上で、
ローンチ戦略を提案してほしい。
```

これだけだ。エージェントはコードベースを読む権限を持っているので、自分でファイルを探索して現状を把握する。

### AIの出力：イシュー定義

```
## イシュー定義

「テンプレート型HP生成SaaSは、Wix/STUDIO等の
 既存プレイヤーが支配する市場で、
 どのセグメントに特化すれば持続的に収益化できるか？」
```

いきなり核心を突いてくる。「どうやって作るか」ではなく「誰に売るか」が本質的な問いだと定義した。これはイシュードリブンの思考法が効いている。

### AIの出力：空（事実）

AIがコードベースを実際に読んで列挙した事実がこちらだ。

```
## 空（事実）

1. 5業種（飲食店、美容院、整体院、歯科医院、士業事務所）の
   テンプレートが実装済み
2. テンプレートは3パターン（シンプル、スタンダード、プレミアム）
3. フォーム入力 → 即時プレビュー → 公開のフローが実装済み
4. 公開後の編集機能が未実装
5. 画像アップロード機能が未実装（プレースホルダー画像のみ）
6. 利用規約・プライバシーポリシーの自動生成機能あり
7. Stripe決済連携は設計段階
8. 独自ドメイン対応は未実装
```

ここがAIコンサルタントの真骨頂だ。人間のコンサルタントならヒアリングに1時間かかる「現状把握」を、AIはコードを直接読んで数分で完了する。しかも「編集機能がない」「画像アップロードがない」といった、開発者本人が見落としがちな欠落を客観的に指摘してくれる。

### AIの出力：雨（解釈）

```
## 雨（解釈）

1. 「作って終わり」のサービスになっている
   → 編集機能がないと解約率が高くなる
2. 5業種×3パターン = 15テンプレートは「狭く深い」戦略
   → 逆にこれが差別化になりうる（Wixは汎用すぎる）
3. 画像なしでは「自分のサイト」感が出ない
   → MVP段階では許容範囲だが、有料化前に必須
4. 利用規約自動生成は独自の強み
   → 小規模事業者が最も面倒に感じる法的文書を解決している
```

### AIの出力：傘（行動）

```
## 傘（推奨アクション）

Phase 1（ローンチ前・2週間）:
  - 画像アップロード機能の実装
  - 公開後の基本編集機能の追加
  - Stripe決済のMVP実装

Phase 2（ローンチ後・1ヶ月）:
  - 飲食店に絞ったマーケティング開始
  - Google マイビジネス連携の検討
  - ユーザーフィードバック収集の仕組み構築

Phase 3（検証後・3ヶ月）:
  - 他業種への展開判断
  - 独自ドメイン対応
  - SEO機能の追加
```

## 価格設定もAIに相談した話

事業分析の延長で、価格設定についても議論した。これが面白い結果になった。

### 最初の提案：3段階プラン

AIは最初、セオリー通りの3段階プランを提案してきた。

```
| プラン     | 月額    | 内容                       |
|-----------|---------|---------------------------|
| ライト    | ¥500    | テンプレ選択・基本公開      |
| スタンダード| ¥1,480  | 編集機能・画像アップロード   |
| プレミアム  | ¥2,980  | 独自ドメイン・SEO・優先サポート|
```

### オーナーの方針を伝えた結果

ここで重要な追加情報を伝えた。

```
課金の動機は「サイトを維持したい」で十分。
松竹梅で迷わせたくない。
1プランでシンプルにしたい。
```

するとAIは推奨を180度変えた。

```
## 推奨案の変更

オーナーの「課金動機はサイト存続で十分」という方針を踏まえると、
1プラン制が最適である。

### 価格比較

¥980/月の場合:
  - 損益分岐点: サーバー費¥5,000/月 → 6ユーザーで黒字
  - 心理的ハードル: 低い（ラーメン1杯以下）
  - 解約率: 低くなりやすい

¥1,480/月の場合:
  - 損益分岐点: 4ユーザーで黒字
  - 心理的ハードル: やや高い（「自分で作れるかも」と比較される）
  - 解約率: ¥980より高くなる可能性

推奨: ¥980/月
理由: ターゲットが小規模事業者（飲食店等）であり、
IT投資への感度が低い層。価格の安さが継続率に直結する。
```

AIは最初セオリー通りの提案をしたが、オーナーの方針という「制約条件」を追加した途端、分析の軸が変わった。これがAIコンサルタントの正しい使い方だ。**AIに考えさせる前に、自分の判断軸を明確に伝える**。

## エージェントの限界と使い所

3ヶ月ほど使ってみて、AIコンサルタントの得意・不得意がはっきりしてきた。

### 「空（事実）」の精度は非常に高い

AIはコードベースを直接読めるため、事実の把握が正確だ。人間のコンサルタントが「御社のシステムの現状を教えてください」とヒアリングする工程を、AIは数分で完了する。しかも「編集機能がない」「画像アップロードが未実装」といった事実を、忖度なしで列挙してくれる。

### 「雨（解釈）」は人間の補正が必要

事実から何を読み取るかは、ドメイン知識に依存する。たとえばAIは「編集機能がないと解約率が上がる」と解釈したが、実際には「飲食店オーナーはそもそも編集しない。メニュー写真を1回載せたら放置する人が多い」というドメイン知識があれば、解釈が変わる。

AIの解釈は「一般論としては正しいが、特定の市場では外れる」ことがある。ここは人間が補正する領域だ。

### 「傘（行動）」は実行可能性を人間が判断する

AIは「Google マイビジネス連携を検討せよ」と提案するが、そのAPI連携にどれだけの工数がかかるか、審査にどれだけ時間がかかるかは考慮していない。行動の優先順位付けは人間の仕事だ。

### 使い所のまとめ

| 用途 | AIの精度 | 備考 |
|------|---------|------|
| 現状把握（事実の列挙） | 高い | コードを直接読めるのが強み |
| 競合分析 | 中程度 | Web検索の精度に依存 |
| フレームワーク適用 | 高い | MECE、空雨傘などを正確に適用する |
| 価格設定の提案 | 中程度 | 制約条件を正しく伝えれば実用的 |
| 市場規模の推定 | 低い | 数字の根拠が曖昧になりがち |
| 実行計画の策定 | 中程度 | 工数見積もりは人間が補正する |

## 並列実行で時間短縮

Claude Codeでは複数のエージェントを同時に起動できる。これを活用すると分析時間を大幅に短縮できる。

たとえば、ローンチ前の最終チェックでこういう使い方をした。

```
以下を並列で実行して:
1. consultantエージェントでビジネス面の最終チェック
2. securityエージェントでセキュリティ監査
3. deployerエージェントでデプロイ設定の確認
```

1つずつ順番に実行すると合計20分以上かかる作業が、並列実行で10分程度に短縮された。それぞれのエージェントが独立したコンテキストで動作するため、互いに干渉しない。

ただし注意点もある。並列で動かすと全エージェントの出力を同時に処理するため、トークン消費量は増える。コスト効率と時間効率のトレードオフだ。

## エージェント定義ファイルのテンプレート

最後に、自分のプロジェクトで使えるコンサルタントエージェントのテンプレートを載せておく。

```markdown
---
name: consultant
description: 事業分析・戦略提案を行うコンサルタントエージェント
tools:
  - Read
  - Grep
  - Glob
  - WebSearch
---

あなたは企業コンサルタントである。

# 分析フレームワーク

## 空・雨・傘
- **空（事実）**: コードベース・ドキュメントから読み取れる客観的事実
- **雨（解釈）**: 事実が事業にとって意味すること
- **傘（行動）**: 具体的な推奨アクション

## 出力ルール
- イシューを1文で定義してから分析を始めること
- 「どれも良い」は禁止。必ず1つの推奨案を根拠付きで提示すること
- 全ての提案が 事実→解釈→行動 でトレーサブルであること
```

これをベースに、自分の事業ドメインに合わせてカスタマイズしていくのが良い。

## まとめ

- Claude Codeのエージェント機能で、McKinsey流のコンサルタントAIを作れる
- `.claude/agents/consultant.md` にフレームワークと出力フォーマットを定義するだけで動く
- AIは「空（事実）」の把握が得意。コードを直接読んで現状を正確に列挙する
- 「雨（解釈）」と「傘（行動）」は人間のドメイン知識で補正が必要
- オーナーの方針や制約条件を明確に伝えることで、AIの提案精度が上がる
- 複数エージェントの並列実行で、分析時間を半分に短縮できる

人間のコンサルタントに依頼すれば数十万円かかる事業分析を、月額のAPI利用料だけで何度でも回せる。精度は人間に劣る部分もあるが、「壁打ち相手」としては十分すぎるほど優秀だ。
